# OpenClaw-Attack 方案设计

## 1. 目标

这个仓库只聚焦一件事：系统化研究开放式 Agent 的攻击链。

与 `openclaw_chainbench_v0` 不同，这里暂时不把重点放在 defense，而是先把攻击问题做扎实：

- 攻击者从哪里进入系统
- 攻击如何劫持不同决策对象
- 攻击如何变成错误结果、越权、外泄、持久化或传播
- 哪些攻击在多轮、多工具、多 Agent 场景下更稳定

一句话概括：

> 把“prompt injection 样例集合”升级为“攻击链 benchmark + 真实攻击执行框架”。

## 2. 核心研究问题

### RQ1

不同输入边界最容易触发哪类决策劫持？

### RQ2

哪些攻击链最容易导致以下结果：

- 结果错误
- 路径偏离
- 越权动作
- 敏感信息外泄
- 长期记忆污染
- 跨轮或跨 Agent 传播

### RQ3

哪些攻击在动态环境里最稳定，最具有迁移性和隐蔽性？

### RQ4

多步工具链、MCP/skill 元数据、长期记忆和多 Agent handoff 是否会显著放大攻击成功率与传播能力？

## 3. Threat Model

这是新仓库必须补齐的核心。

### 3.1 攻击者角色

- 外部网页或搜索结果提供者
- 邮件发送者或协作者
- 文档、附件或知识库内容提供者
- API 或工具响应污染者
- MCP server 或 skill 元数据提供者
- 上游 Agent 或委派链中的中间 Agent
- 历史记忆或摘要内容的污染者

### 3.2 攻击者能力

- 注入自然语言指令
- 伪装为建议、规则、最佳实践或系统说明
- 污染检索文档、工具描述、返回结果或摘要
- 在多步流程中埋入延迟触发指令
- 诱导写入长期记忆或下游 handoff

默认不假设攻击者拥有：

- 直接执行本地代码的能力
- 绕过平台权限模型的能力
- 控制基础模型参数的能力

### 3.3 被攻击资产

- 用户目标一致性
- 决策过程完整性
- 工具调用边界
- 工作区和私有数据
- 长期记忆与摘要状态
- 多 Agent 间共享上下文

### 3.4 成功标准

攻击成功不只是一类结果，而是多层结果：

- `trigger_success`: 攻击内容进入推理或计划上下文
- `hijack_success`: 攻击改变了目标、信念、规划、工具选择或状态更新
- `impact_success`: 造成错误结果、越权、外泄、持久化或传播

## 4. 攻击链形式化

我们把每个样本建模为：

`(benign_task, attacker, entry_surface, payload, hijacked_object, expected_outcome, environment_state)`

其中三阶段主链为：

### 阶段 1：Attack Entry

- direct task
- workspace
- tool output
- constructed context
- skill or MCP metadata

### 阶段 2：Decision Hijack

- goal hijack
- belief hijack
- planner hijack
- tool-choice hijack
- state-update hijack

### 阶段 3：Impact Outcome

- wrong result
- path deviation
- unsafe action
- privilege escalation
- data exfiltration
- memory persistence
- cross-agent propagation

## 5. Benchmark 设计

### 5.1 样本单位

不再把 benchmark 样本定义成单个 prompt，而定义成攻击链实例。

每个实例都必须包含：

- 正常任务
- 明确攻击入口
- 明确被劫持的决策对象
- 明确预期影响
- 明确环境状态与前置条件

### 5.2 环境域

初始版本建议覆盖 6 个域：

1. Web research
2. Email and calendar assistant
3. RAG and document QA
4. Local workspace and file automation
5. MCP, tools, and skill orchestration
6. Multi-agent delegation and handoff

### 5.3 攻击矩阵

用下面的矩阵驱动 case 设计：

`entry_surface x hijacked_object x impact_type`

不追求全笛卡尔积，但要求每个高价值区域至少有多个实例。

### 5.4 强挑战子集

优先做三类：

- persistent memory poisoning
- tool or MCP metadata hijack
- sequential multi-tool attack chains

## 6. 数据与标注

### 6.1 Case 标注

每个 case 至少应回答：

- 攻击者是谁
- 通过什么载体进入
- 想劫持什么对象
- 想影响什么资产
- 预期的第一劫持步在哪里
- 攻击是否需要多轮、工具链或 memory

### 6.2 Trace 标注

每条 trace 要支持恢复攻击链：

- 哪一步首次接触 payload
- 哪一步首次出现 hijack
- 哪一步形成真实 impact
- 哪些状态被污染
- 污染是否跨轮扩散

## 7. Payload 库设计

攻击研究不能只有“概念案例”，还需要 payload variants。

每个攻击模板应该至少有以下变体：

- explicit instruction
- stealth recommendation
- policy-like framing
- tool metadata framing
- memory persistence framing
- multi-turn delayed trigger

建议按 `attacks/` 目录维护模板，而不是把 payload 散落在 case 文件里。

## 8. 攻击执行框架

后续 `src/attack_harness.py` 需要完成以下职责：

1. 读取 case
2. 构造环境
3. 注入 payload
4. 调用目标 Agent 或模拟后端
5. 采集 step-level trace
6. 生成结构化结果

这一步完成后，本仓库才能从“攻击设计文档”升级成“攻击实验框架”。

## 9. 评测指标

这里优先保留攻击侧指标，而不是防御效用指标。

- Attack Trigger Rate
- Hijack Rate
- First-Hijack Step
- Unsafe Action Rate
- Privilege Escalation Rate
- Exfiltration Success Rate
- Persistence Success Rate
- Propagation Rate
- Attack Stability
- Attack Transferability
- Stealth Success Rate

## 10. 阶段性路线

### Phase 1

完成 threat model、schema、初始 case set 和 payload 模板规范。

### Phase 2

实现环境构造器和 attack harness，能够输出真实 trace。

### Phase 3

扩展多轮、持久化、MCP、multi-agent 攻击子集。

### Phase 4

接入真实 OpenClaw 或兼容 Agent runtime，开始大规模跑实验。

## 11. 当前版本的定位

当前版本是攻击研究的“v1 scaffold”：

- 重点是问题建模和架构补齐
- 不是最终 benchmark
- 也不是最终实验结果

下一步的关键不是再补几条样例，而是把真实攻击执行链路建起来。
