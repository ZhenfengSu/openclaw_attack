# 攻击缺口分析

## 1. 当前最缺的东西

如果只考虑攻击而不考虑防御，当前项目最缺的是四类基础能力：

1. 明确的 threat model
2. 系统化的攻击覆盖矩阵
3. 真实攻击执行与 trace 采集能力
4. 可复用的 payload 变体库

缺少这四项时，仓库仍然更像“攻击设想集合”，而不是“攻击研究框架”。

## 2. 具体缺口

### 2.1 Threat Model 缺口

目前必须明确而尚未完全固定的内容包括：

- 攻击者身份与位置
- 攻击者能控制哪些输入面
- 是否能控制排序、检索结果、metadata 或 memory
- 攻击目标是误导、越权、外泄还是长期控制
- 成功标准如何分层定义

### 2.2 Coverage 缺口

当前攻击面覆盖仍不完整，尤其缺少：

- 数据外泄型攻击
- 显式权限升级链
- 多跳工具组合攻击
- 反馈回路与自强化攻击
- 配置、角色、系统说明污染
- 伪装成正常策略或最佳实践的隐蔽攻击

### 2.3 Payload 缺口

当前缺的不是“有没有攻击想法”，而是：

- 同类攻击的多表述变体
- 同一目标在不同载体上的迁移版本
- stealth vs explicit 的强弱版本
- delayed trigger 的多轮版本
- 持久化与传播导向版本

### 2.4 Execution 缺口

最关键的工程缺口是：

- 没有真正把攻击注入运行环境的 harness
- 没有自动生成环境状态的 builder
- 没有把真实执行过程导出为标准 trace
- 没有把攻击结果和 benign reference 进行程序化对比

### 2.5 Measurement 缺口

当前缺少更偏攻击侧的量化维度：

- 触发率
- 首次劫持步
- 攻击链长度
- 隐蔽成功率
- 外泄成功率
- 持久化成功率
- 传播深度
- 攻击稳定性
- 攻击迁移性

## 3. 优先级建议

### P0

- 固化 threat model
- 固化 case schema
- 补覆盖矩阵

### P1

- 建 payload 模板库
- 建环境构造器
- 建 trace schema 与导出器

### P2

- 接入真实 Agent backend
- 跑多轮和多 Agent 攻击
- 做稳定性与迁移性分析

## 4. 新仓库要解决什么

`openclaw_attack` 的职责应该非常清晰：

- 为攻击研究提供统一 schema
- 为攻击链 benchmark 提供 case 组织方式
- 为攻击执行提供环境与 payload 抽象
- 为攻击分析提供 trace 和指标基础

如果后续要和 defense 仓库结合，最好把关系定义成：

- `openclaw_attack`: 负责攻击设计、执行和测量
- `openclaw_chainbench_v0`: 负责 defense prototype 与对比实验
